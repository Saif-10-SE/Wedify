Context Diagram
The scope [1] description establishes the boundary and connections between the system youâ€™re developing and everything else in the universe. The context diagram visually illustrates this boundary. It identifies external entities (also called terminators) outside the system that interface to it in some way, as well as data, control, and material flows between the terminators and the system. The context diagram is the top level in a data flow diagram developed according to the principles of structured analysis.
Figure below illustrates a portion of the context diagram for the Chemical Tracking System. The entire system is depicted as a single circle; the context diagram deliberately provides no visibility into the systemâ€™s internal objects, processes, or data. The â€œsystemâ€ inside the circle could encompass any combination of software, hardware, and human components. Therefore, it could include manual operations as part of the entire system. The external entities in the rectangles can represent user classes (Chemist, Buyer), organizations (Health and Safety Department), other systems (Training Database), or hardware devices (Bar Code Reader). The arrows on the diagram represent the flow of data (such as a request for a chemical) or physical items (such as a chemical container) between the system and its external entities.1

DFDs[1] provide a big-picture view of how data moves through a system, which other models donâ€™t show well. Various people and systems execute processes that use, manipulate, and produce data. Other diagram canâ€™t show you the full life cycle of a piece of data. Multiple pieces of data might be pulled together and transformed by a process (for example, shopping cart contents plus shipping information plus billing information are transformed into an order object).

Figure 2: Data Flow Diagram - Chemical Tracking System[1]

The single circle that represented the entire Chemical Tracking System on the context diagram has been subdivided into six major processes (the process bubbles). As with the context diagram, the external entities are shown in rectangles. All data flows (arrows) from the context diagram also appear on the level 0 DFD. In addition, the level 0 diagram contains several data stores, depicted as a pair of parallel horizontal lines, which are internal to the system and therefore do not appear on the context diagram. A flow from a bubble to a store indicates that data is being placed into the store, a flow out of the store shows a read operation, and a bidirectional arrow between a store and a bubble indicates an update operation.

Following are important concepts related to Product backlog:


Figure 3: Epic vs User Story in Product Backlog [1]


Sprint:
In Agile project management, a sprint is a fixed-length period during which a team works to complete a set of tasks and deliver a potentially shippable increment of product. Sprints are typically one to four weeks long, but two to three weeks is a common duration.
Epic:
An epic is a user story that is too large to fully implement in a single iteration. Its a multi-sprint project goal or a broad area of functionality that requires significant development effort.  Because epics span iterations, they must be split into sets of smaller stories. Sometimes epics are large enough that they must be subdivided into multiple epics, each of which is then split into multiple stories until each resulting story can be reliably estimated and then implemented and tested within a single iteration. Breaking epics down into smaller epics and then into user stories is often referred to as story decomposition  
Scope:
Covers multiple releases or iterations and often spans across several teams. 
Purpose:
Provides a high-level overview of the project and helps in prioritizing and planning the release. 
Example:
"Implement e-commerce functionality". 
Feature:
A specific piece of functionality or a deliverable within an Epic that provides value to the user. 
Scope:
Smaller than an Epic and can be developed and released in a single iteration. 
Purpose:
Breaks down the Epic into manageable chunks and guides the development process. 
Example:
"Product listing page," "Order page," "Payment page" within the "e-commerce functionality" Epic. 
User Story:
A user story is a â€œshort, simple description of a feature told from the perspective of the person who desires the new capability, usually a user or customer of the systemâ€. User stories are sized so as to be fully implementable in a single iteration. Each completed user story iteratively builds the feature. User stories often are written according to the following template, although other styles also are used:
As a <type of user>, I want <some goal> so that <some reason OR Benefit>. 
Scope:
The smallest unit of work that can be delivered within a sprint. 
Purpose:
Captures the requirements of a feature in a way that is easy for developers to understand and implement. 
Example:
"As a user, I want to add items to my shopping cart so that I can buy them later"


Acceptance Criteria
A set of conditions or rules that a user story must satisfy to be considered "done" and accepted by the product owner.
Purpose:
To define what success looks like for a user story.
Written by:
Usually written by the product owner, sometimes with input from QA and developers.
Format:
Often in plain language or BDD style (Given-When-Then).
Example:
User Story: As a user, I want to reset my password so I can access my account if I forget it.
Acceptance Criteria:
The user must be able to request a password reset by entering their email.
The system must send a reset link if the email is registered.
The reset link must expire in 24 hours.
The user must be able to set a new password via the reset link.

Story Point:
In Agile, story points are a relative unit of measurement used to estimate the effort required to complete a user story or other piece of work. They are not time-based but rather represent the complexity and effort relative to other items on the backlog.
Key aspects of story points range in Agile:
Relative, not absolute:
Story points are assigned relative to other tasks, meaning a story with 2 points is roughly twice as complex as a story with 1 point. 
No specific time value:
Story points don't directly equate to hours or days; they represent a degree of effort or complexity. 
Common scales:
Teams often use scales like Fibonacci (1, 2, 3, 5, 8, 13...)
Benefits:
Story points help teams estimate work, plan sprints, and manage expectations, rather than focusing on individual time estimates.

Backlog grooming:
It is a process where the product owner and development team collaboratively review and refine the product backlog. It is a crucial practice in Agile project management that involves regularly updating and refining the product backlog. It focuses on ensuring the backlog is well-defined, prioritized, and ready for upcoming sprint cycles, ultimately contributing to a more efficient and effective development process.

Refining user stories: Making sure they are clear, concise, and easy to understand.

Sprint 1 Decision Table
A software system is often governed by complex logic, with various combinations of conditions leading to different system behaviors. In such kind of user stories, it is advisable to create decision table in order to better understand possible combinations. Tesers are also advised to develop a separate test case for each possible combinations that check whether system is working fine in all those complex possible combinationss of input and outputs.
A sample Figure below shows a decision table for the logic that governs whether the Chemical Tracking System should accept or reject each request for a new chemical. Four factors influence this decision:
Whether the user who is creating the request is authorized to request chemicals
Whether the chemical is available either in the chemical stockroom or from a vendor
Whether the chemical is on the list of hazardous chemicals that require special training in safe handling
Whether the user who is creating the request has been trained in handling this type of hazardous chemical.

ERD
We now present another example, a UNIVERSITY database, [3] to illustrate the ER modeling concepts. Suppose that a database is needed to keep track of student enrollments in classes and studentsâ€™ final grades. After analyzing the miniworld rules and the usersâ€™ needs, the requirements for this database were determined to be as follows (for brevity, we show the chosen entity type names and attribute names for the conceptual schema in parentheses as we describe the requirements; relationship type names are only shown in the ER schema diagram):
The university is organized into colleges (COLLEGE), and each college has a unique name (CName), a main office (COffice) and phone (CPhone), and a particular faculty member who is dean of the college. Each college administers a number of academic departments (DEPT). Each department has a unique name (DName), a unique code number (DCode), a main office (DOffice) and phone (DPhone), and a particular faculty member who chairs the department. We keep track of the start date (CStartDate) when that faculty member began chairing the department.
A department offers a number of courses (COURSE), each of which has a unique course name (CoName), a unique code number (CCode), a course level (Level: this can be coded as 1 for freshman level, 2 for sophomore, 3 for junior, 4 for senior, 5 for MS level, and 6 for PhD level), a course credit hours (Credits), and a course description (CDesc). The database also keeps track of instructors (INSTRUCTOR); and each instructor has a unique identifier (Id), name (IName), office (IOffice), phone (IPhone), and rank (Rank);
in addition, each instructor works for one primary academic department.
The database will keep student data (STUDENT) and stores each studentâ€™s name (SName, composed of first name (FName), middle name (MName), last name (LName)), student id (Sid, unique for every student), address (Addr), phone (Phone), major code (Major), and date of birth (DoB). A student is assigned to one primary academic department. It is required to keep track of the studentâ€™s grades in each section the student has completed.
Courses are offered as sections (SECTION). Each section is related to a single course and a single instructor and has a unique section identifier (SecId). A section also has a section number (SecNo: this is coded as 1, 2, 3, . . . for multiple sections offered during the same semester/year), semester (Sem), year (Year), classroom (CRoom: this is coded as a combination of building code (Bldg) and room number (RoomNo) within the building), and days/times (DaysTime: for example, â€˜MWF 9am-9.50amâ€™ or â€˜TR 3.30pm-5.20pmâ€™â€” restricted to only allowed days/time values). (Note: The database will keep track of all the sections offered for the past several years, in addition to the current offerings. The SecId is unique for all sections, not just the sections for a particular semester.) The database keeps track of the students in each section, and the grade is recorded when available (this is a many-to-many relationship between students and sections). A section must have at least five students.

The ER diagram for these requirements is shown in Figure 28 using the min-max ER diagrammatic notation. Notice that for the SECTION entity type, we only showed SecID as an underlined key, but because of the miniworld constraints, several other combinations of values have to be unique for each section entity. For example, each of the following combinations must be unique based on the typical miniworld constraints:
(SecNo, Sem, Year, CCode (of the COURSE related to the SECTION)): This specifies that the section numbers of a particular course must be different during each particular semester and year.
(Sem, Year, CRoom, DaysTime): This specifies that in a particular semester and year, a classroom cannot be used by two different sections at the same days/time.
(Sem, Year, DaysTime, Id (of the INSTRUCTOR teaching the SECTION)): This specifies that in a particular semester and year, an instructor cannot teach two sections at the same days/time. Note that this rule will not apply if an instructor is allowed to teach two combined sections together in the particular university
Data Dictionary
A data dictionary [1] is a collection of detailed information about the data entities used in an application. Collecting the information about composition, data types, allowed values, and the like into a shared resource identifies the data validation criteria, helps developers write programs correctly, and minimizes integration problems. The data dictionary complements the project glossary, which defines application domain or business terms, abbreviations, and acronyms. We recommend keeping the data dictionary and the glossary separate.This information feeds into design in the form of database schemas, tables, and attributes, which ultimately lead to variable names in programs. The time you invest in creating a data dictionary will be more than repaid by avoiding the mistakes that can result when project participants have different understandings of the data.
